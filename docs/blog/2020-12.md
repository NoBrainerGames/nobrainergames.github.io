<sup>First 3D avatar in Nim.</sup>

This scene was the product of the addition of two systems: (1) the physics engine and (2) an entity component system. The first system established gravity in the scene, allowing the avatar to "fall" to the floor. The second system enabled navigation using the mouse.

The physics system wrapped AMD's [FEMFX](https://gpuopen.com/femfx/) library, which was a newly opensourced project at the time. I considered going with Nvidia's [PhysX](https://www.nvidia.com/en-us/drivers/physx/physx-9-19-0218-driver/) library initially, but after some research I concluded that I should go with a smaller library that I could easily contribute to if I ever needed to add a custom feature.

The entity component system (ECS) implementation was inspired by skypjack's amazing [series on ECS](https://skypjack.github.io/2019-02-14-ecs-baf-part-1/). I decided to implement a pure Nim version instead of wrapping his library to better take advantage of Nim's language features.

It was here that I first conceptualized the idea of an "implicit event system". The idea is that the engine should be able to notify user-defined systems of mutations to component fields without explicitly invoking any callback functions.

To illustrate, take two systems defined in the files `players.nim` and `followers.nim`. An "event" can be defined as a mutation of any field of a component handled by a system, e.g. the `Player` component in `players.nim`. This event can be seen by another system that also handles the `Player` component type simply by defining

```nim
# players.nim: system of players
proc onUpdate() =
  player.position = (2, 4) # move player to coordinates (x, y)
```

```nim
# followers.nim: system of followers
proc onMutate(position) = # automatically called from 'players.nim'
  follower.position = position # move follower to player's location
```
